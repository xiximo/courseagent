1. 产品定位与核心原则

本产品是一款 AI 课程顾问 Agent，部署于 AI 教育中心暑期咨询场景，替代部分人工客服工作，解决信息回复不一致、班型推荐缺少依据、课程价格与平台会员价格混淆等问题。

身份未明，不推荐班型
角色与资料一一对应，不跨库混用
课程价 ≠ 平台会员价
LLM 动态生成，禁止硬编码答案
设计目标：在官方三份资料（素材 A/B/C）的知识边界内，通过结构化对话流程，为三类用户提供准确、可追溯、可验收的咨询服务。

产品形态（v1.1 补充）

开放接入端（无授权）

面向 C 端访客：独立对话页、JS 嵌入挂件、REST API。无需登录即可咨询，通过 agentId / embedKey 识别租户与配置。

SaaS 管理端（需授权）

面向机构运营：登录后台配置模型、状态机、知识库、欢迎语与嵌入代码。配置变更实时生效，无需改代码发版。

2. 业务背景与角色

角色 典型诉求 Agent 应完成的工作
学生 / 家长 选择城市、日期、线上或线下班型，咨询费用与准备事项 识别学生端需求，采集约束，推荐 1—2 个班型并基于 素材 A 回答
教师 选择暑期集训或周末研修，咨询培训目标、安排和报名 识别教师端需求，采集时间偏好并基于 素材 B 推荐
机构 / 企业人员 了解 OPC 平台服务、会员权益或企业合作 仅基于 素材 C 介绍平台服务，不将其价格当作课程费用

3. 知识边界与资料路由

资料 内容范围 允许用途 禁止用途
素材 A
暑期 AI 素养夏令营手册		 北京线下班、上海线下班、线上直播班 学生课程详情、比较和推荐 不得回答教师培训或平台会员价格
素材 B
教师 AI 素养培训体系介绍		 暑期集训班、周末研修班 教师课程详情、比较和推荐 不得回答学生营期或平台服务价格
素材 C
OPC 平台产品白皮书		 平台、会员、企业合作、定制服务 机构与平台类咨询 不得参与学生/教师班型推荐，不得替代课程费用

系统层硬规则（不仅依赖 Prompt）

  • 检索范围 = 当前 role 绑定的唯一知识库，禁止跨库检索
  • 用户询问「另一侧」价格或班型时，明确拒绝并引导回菜单
  • 平台会员价与课程价分字段、分文档存储，回复前做角色 + 关键词校验
  • 资料外信息一律回复「无法确认」，不得编造

4. 七步业务流程（状态机）

1 欢迎分流 → 2 身份澄清 → 3 约束采集 → 4 推荐 → 5 详情追问 → 6 报名引导

步骤 7「会话控制」可在任意阶段触发：取消 / 重来 / 重新开始 / 查看所有课程

步骤 触发条件 系统行为 可观察结果
1 欢迎与分流 新会话 说明服务范围，提供学生课程、教师培训、平台服务三个入口（按钮 + 自然语言） 用户进入对应分支
2 身份澄清 身份未明确 询问是学生/家长、教师，还是机构人员 未澄清身份前不得直接推荐班型
3 约束采集 用户要求推荐 依次获取城市/地点、可用日期、线上线下偏好、学习目标；已提供的信息不重复询问 形成至少两个有效约束，或说明资料不足
4 推荐 约束足够 从对应核心资料推荐 1—2 个真实班型，给出名称、关键安排和逐条理由 推荐与身份及约束一致，不出现资料外班型
5 详情追问 用户追问费用、时间、地点、师资、大纲、物资或报名 关联当前班型检索回答，末尾标注文档名称和章节 事实与资料一致，引用可定位
6 报名引导 用户表达报名意向 仅提供资料中已有的报名方式；资料未提供时说明需人工确认 不虚构链接、电话、余位或截止状态
7 会话控制 取消/重来/重新开始/查看所有课程 清空上下文或列出当前身份下全部班型 重置后不继续引用旧班型

会话状态数据结构

SessionState { sessionId: string step: "welcome" | "identity" | "constraints" | "recommend" | "qa" | "enroll" | "reset" role: "student" | "teacher" | "org" | null constraints: { city?: string date?: string format?: "online" | "offline" goal?: string } recommendedCourses: Course[] // 当前推荐的 1-2 个班型 lockedCourse?: Course // 多轮追问时锁定的班型 messageHistory: Message[] }

4.1 状态机 vs ReAct 选型分析

结论：采用 「显式状态机编排 + 分步 LLM + RAG」 混合架构。状态机负责流程、门禁与资料路由（确定性）；LLM 负责约束抽取、推荐文案与详情问答（动态生成）。ReAct 仅可作为步骤内可选增强，不宜作为全流程主编排器。

需求对架构的三类约束

约束类型 需求体现 更适合的实现
流程约束 七步顺序、先澄清再推荐、约束足够才推荐、重置后清空上下文 显式状态机（FSM）
边界约束 素材 A/B/C 不混用、课程价≠会员价、资料外不编造 系统层规则 + 状态门禁
生成约束 推荐/问答须 LLM 结合资料动态生成，禁止关键词硬编码整段答案 特定步骤内调用 LLM + RAG

需求中的「Agent」指业务角色（课程顾问），并非要求必须使用 ReAct 类自主规划框架。赛题本质是 强流程 + 强边界 + 强验收 的客服场景，而非开放式工具调用任务。

方案对比（结合官方验收）

需求点 显式状态机 纯 ReAct Agent
七步流程逐步推进 ✓ 天然匹配 步骤可枚举、跳转可预测 ✗ 易跳步 模型可能跳过欢迎/身份直接推荐
未澄清身份不得推荐 ✓ 硬门禁 role==null → block ✗ 不稳定 仅靠 Prompt 约束，易越权
约束依次采集、不重复问 ✓ 缺啥问啥 constraints 对象驱动 ✗ 易重复/遗漏 依赖模型记忆
至少 2 个约束才推荐 ✓ 计数判断 确定性 transition △ 不稳定 同一输入结果可能不同
素材 A/B/C 不混用 ✓ 检索层硬限 role → kb 绑定 ✗ 易混库 Tool 选择不可控
多轮继承班型上下文 ✓ lockedCourse session 显式存储 △ 易丢失 依赖上下文窗口
25 条验收用例 ≥90% ✓ 可复现 同输入同路径 ✗ 波动大 非确定性难稳定过测
SaaS 后台可视化编排 ✓ 节点+连线 运营可配置 ✗ 难可视化 只能调 Prompt/Tool
LLM 动态生成推荐/问答 ✓ 分步调用 状态机决定何时调 LLM ✓ 可以 但需额外流程控制

纯 ReAct 的主要风险

  • 流程越权：用户说「推荐个便宜的」时，可能跳过身份澄清与约束采集
  • 边界失守：Thought 阶段自行决定检索库，学生场景可能答出平台会员价
  • 不可复现：评委连测 25 条用例时，结果波动导致功能/知识维度扣分
  • 成本更高：每轮多次 LLM 调用（Thought → Action → Observation 循环）
  • 评分红线：身份/资料边界错误可致需求解读 0—7 分；硬编码补救又可能触发技术实现 0—11 分

推荐架构：State Machine Orchestrated Agent

┌─────────────────────────────────────────────────────────┐ │ 状态机（主编排器 · 确定性） │ │ step / role / constraints / lockedCourse / 跳转规则 │ └───────────────────────────┬─────────────────────────────┘ │ 按 step 分发 ┌───────────────────┼───────────────────┐ ▼ ▼ ▼ 固定模板 LLM 抽取 LLM + RAG (welcome/reset/ (NL→constraints) (recommend/qa/enroll) error/menu) + Guardrails

各步骤：编排 vs 生成

步骤 状态机（编排） LLM（生成）
1 欢迎分流 固定菜单 + 按钮入口；NL 识别后 transition 可选：NL 意图 → role 分类
2 身份澄清 role 为空则停留，禁止进入 recommend 识别「我是家长/教师/机构」
3 约束采集 按缺字段列表决定下一问；已采集不重复 从自然语言抽取 city/date/format/goal
4 推荐 有效约束 ≥2 才 transition；锁定 recommendedCourses + RAG 生成 1—2 班型及逐条理由
5 详情追问 必须存在 lockedCourse；继承班型上下文 + RAG 回答并强制标注引用章节
6 报名引导 仅响应 enroll 意图；无资料则固定「人工确认」 + RAG 提取报名方式（有则展示）
7 会话控制 reset 清空 state；list 列出当前 role 全部班型 不使用（固定模板）

状态转移表

当前状态 条件 下一状态 动作
welcome 用户选择入口 / 识别到 role identity 或 constraints 展示欢迎语；若 role 已明确可跳过 identity
identity role 已明确 constraints 询问或确认身份
identity 用户要求推荐但 role 为空 identity（保持） 拦截推荐，继续澄清身份
constraints 有效约束 < 2 constraints（保持） 追问缺失字段
constraints 有效约束 ≥ 2 recommend LLM + RAG 生成推荐，写入 lockedCourse
recommend / qa 用户追问细节 qa 基于 lockedCourse + RAG 回答
qa 用户表达报名意向 enroll 展示资料中报名方式或提示人工确认
任意 「重新开始 / 取消 / 重来」 welcome 清空 role/constraints/lockedCourse
任意 「查看所有课程」 —（保持） 列出当前 role 下全部班型，不切换 step

ReAct 的适用边界（可选增强）

ReAct 适合工具多、路径开放、允许试错的任务（如代码生成、多源调研）。在本赛题中，若使用 ReAct，建议仅限步骤 5「详情追问」内部：

用户：「这个班的师资怎么样？」 Thought → 需要查 lockedCourse 的师资信息 Action → search_kb(kb=role对应库, course=lockedCourse, topic=师资) Observation → 检索片段 Answer → LLM 生成回答 + 引用章节
  • Tool 集合固定（search_kb、list_courses），不允许 Agent 自行切换 role 或跳转步骤
  • 这是 Step-scoped ReAct，不是 Global ReAct
  • 48 小时冲刺阶段非必须，优先保证状态机主链路稳定

选型定案

方案 定位 建议
显式状态机 主编排器 采用 负责七步流程、门禁、上下文、资料路由
LLM + RAG 内容生成器 采用 负责推荐、问答、约束抽取（满足动态生成要求）
Step-scoped ReAct 步骤内增强 可选 仅用于详情追问的多工具检索
纯 ReAct 替代状态机 全局编排 不推荐 难满足验收与边界，评分风险高
产品命名与实现：对外可称「AI 课程顾问 Agent」，技术实现应为 State Machine Orchestrated Agent —— 状态机定规矩，LLM 填内容，Guardrails 守边界。

请求处理时序

  1. 用户输入 → 输入校验（空/超长/特殊符号）
  2. 意图识别 → 判断当前步骤应执行的操作
  3. 身份门禁 → 未澄清则拦截推荐类请求
  4. 按 role 检索对应知识库片段
  5. 组装 Prompt（状态 + 检索片段 + 输出规范）→ 调用 LLM
  6. 后处理校验 → 补充/修正引用 → 返回用户

7. 对话与 Prompt 设计

System Prompt 结构

  1. 角色定义：你是 AI 教育中心的课程顾问，只能基于提供的资料回答问题
  2. 当前上下文注入：用户身份、已采集约束、当前锁定班型（由状态机注入）
  3. 可用资料片段：检索到的相关内容（限定在当前 role 的知识库）
  4. 输出规范：推荐 1—2 个班型；理由逐条对应约束；引用格式统一
  5. 禁止项：编造信息、跨资料混用、虚构报名方式、混淆价格

引用格式规范

每条事实性回答末尾标注：
来源：《素材A·暑期AI素养夏令营手册》第3章「北京线下班安排」

三类 LLM 调用场景

推荐生成

输入：身份 + 约束 + 检索到的班型资料
输出：1—2 个班型名称、关键安排、逐条理由

详情问答

输入：锁定班型 + 用户问题 + 检索片段
输出：事实回答 + 文档章节引用

报名引导

输入：锁定班型 + 报名相关检索
输出：资料中已有的报名方式，或说明需人工确认

固定模板（无需 LLM）

  • 欢迎语与菜单
  • 重置成功提示
  • 空输入 / 超长 / 模型异常提示
  • 无关问题边界说明

8. 界面与交互设计

产品包含两类界面：开放接入端（C 端访客，无需登录）与 SaaS 管理端（机构运营，需授权）。二者共用同一 Agent 引擎与配置数据。

开放接入端 — 页面布局

开放接入端 — 交互要求（对齐验收）

  • 至少一个可交互界面，用户消息与 AI 消息视觉区分
  • 提交后有 Loading 反馈（typing indicator 或 spinner）
  • 错误信息可见（模型失败、输入无效等）
  • 快捷按钮支持三大入口 + 查看全部 + 重新开始
  • 无需登录；通过 URL 参数或 embedKey 绑定 Agent 实例
  • 可选：顶部状态条展示当前身份、约束、锁定班型（便于演示与调试）

14. 开放接入：嵌入与 API

问答交互面对 C 端用户完全无授权：访客打开链接或访问嵌入站点即可对话，不需要注册/登录。安全与隔离通过 agentId、embedKey 和域名白名单实现，而非终端用户身份认证。

三种接入方式

方式 适用场景 集成成本 说明
独立对话页 快速体验、大赛演示、分享链接 零集成 https://chat.example.com/a/{agentId}，扫码或链接直达
JS 嵌入挂件 机构官网、活动页、H5 落地页 一行代码 右下角浮窗或内嵌容器，样式可配置
开放 API 自有 App、小程序、CRM/客服系统 按接口文档开发 REST 同步或 SSE 流式，服务端用 API Key 鉴权

JS 嵌入示例

<!-- 机构网站任意页面底部 --> <script src="https://cdn.example.com/course-agent-widget.v1.js" data-agent-id="agt_xxx" data-embed-key="emb_xxx" data-mode="bubble" <!-- bubble | inline --> data-theme="light" data-position="bottom-right" async ></script> <!-- 或内嵌模式 --> <div id="course-agent-chat"></div> <script> CourseAgent.init({ container: '#course-agent-chat', agentId: 'agt_xxx', embedKey: 'emb_xxx', locale: 'zh-CN' }); </script>

开放 API 设计（概要）

接口 方法 鉴权 说明
/v1/agents/{agentId}/sessions POST embedKey（前端）/ API Key（服务端） 创建匿名会话，返回 sessionId
/v1/sessions/{sessionId}/messages POST sessionId + embedKey 发送用户消息，返回 AI 回复（支持 stream=true）
/v1/sessions/{sessionId}/messages GET sessionId 拉取历史消息（可选）
/v1/sessions/{sessionId}/reset POST sessionId 重置会话，清空上下文
/v1/agents/{agentId}/config GET embedKey 获取欢迎语、快捷按钮、主题等公开配置

安全策略

  • 终端无授权：C 端不传 Token；仅持有合法 embedKey 且来源域名在白名单内的请求可创建会话
  • 服务端 API Key：第三方后端集成时使用，密钥仅存服务端，禁止暴露在前端 JS
  • 限流：按 agentId / IP / sessionId 限流，防刷与滥用
  • CORS：Widget 与 API 仅允许配置的 origin 跨域访问
  • 内容安全：输入校验、输出过滤与 Guardrails 与主流程一致
与大赛验收的关系：48 小时冲刺阶段至少交付「独立对话页」满足可交互界面要求；JS 嵌入与 API 作为产品化扩展与加分项，架构预留接口即可。

配置数据模型（核心）

AgentConfig { agentId: string tenantId: string name: string status: "active" | "disabled" model: { provider: string // openai | deepseek | qwen ... modelName: string apiKeyRef: string // 密钥引用，非明文 params: { temperature, maxTokens, timeout } prompts: { system, recommend, qa, enroll } } stateMachine: { steps: StepDefinition[] // 七步流程定义 transitions: Transition[] // 跳转规则 constraintFields: Field[] // 约束采集字段 templates: Record<string, string> // 固定文案 } knowledgeBases: { role: "student" | "teacher" | "org" kbId: string bindRules: string[] // 禁止跨库规则 }[] embed: { embedKey: string allowedOrigins: string[] theme: { primaryColor, mode, logoUrl } } }

配置生效机制

  1. 管理员在后台保存配置 → 写入 DB + 发布版本号
  2. 运行时引擎启动或收到配置变更事件 → 拉取最新 AgentConfig
  3. 新会话使用新配置；进行中的会话可配置「随重置生效」或「立即切换」策略
  4. 知识库更新后触发异步 re-index，完成后切换检索索引指针
与现有 webadmin 的关系：可复用项目中 webadmin 的 Agent Ops、LLM 配置、知识库同步等模块，扩展「状态机编排」与「嵌入接入」页面，形成完整 SaaS 管理端。

9. 异常与输入处理

场景 触发条件 系统响应 预期行为
空输入 用户发送空白消息 「请输入您的问题」 不崩溃，可继续操作
超长输入 输入超过 500 字 「输入过长，请精简后重新发送（限 500 字）」 不调用 LLM，可继续操作
特殊符号 输入含 <script>、SQL 注入等 安全转义处理，正常响应或边界提示 不崩溃
无关问题 如「今天天气怎么样」 说明服务边界 + 返回菜单选项 不尝试回答，引导回正题
模型异常 LLM API 超时或报错 「服务暂时不可用，请稍后重试」 会话状态保留，可继续操作
跨资料混淆 学生问平台会员价 「平台会员价格属于平台服务范畴，与学生课程费用不同。如需了解请切换至平台服务入口。」 价格不混淆
资料外问题 询问五份资料之外的班型 「该信息不在现有资料范围内，无法确认，建议联系人工客服。」 不编造
多轮对话最低要求：至少支持「推荐 → 追问时间 → 追问准备事项」连续三轮，全程正确继承同一班型上下文。

11.1 问答质量评测方案（LangSmith / Langfuse）

结论：推荐使用 LangSmith（LangChain 官方评测平台）或开源替代 Langfuse 做离线回归评测。采用三层评测：Code Evaluator 守红线 → Reference Match 守事实 → LLM-as-Judge 看语义。48 小时冲刺优先 LangSmith + OpenEvals 快速出报告；长期 SaaS 可考虑 Langfuse 自托管。

工具选型说明

工具 定位 核心能力 本项目建议
LangSmith LangChain 官方 SaaS Dataset、离线/在线评测、实验对比、Trace、evaluate() 48h 首选 上手快、模板全、易导出测试证据
OpenEvals LangChain 开源评测包 预置 LLM-as-Judge（正确性、Faithfulness 等） 与 LangSmith 配合，减少自建 Judge Prompt
AgentEvals LangChain Agent 评测包 轨迹评测、Tool 调用顺序与参数校验 若步骤内使用 ReAct/Tool 时启用
Langfuse 开源（MIT）可自托管 Trace、Code Evaluator、实验对比、Prompt 管理 长期备选 数据主权、CI 回归、控成本

LangSmith / Langfuse 均不强制使用 LangChain 框架，Python / TypeScript 任意栈可通过 SDK 接入。赛题未强制要求评测平台，但作为技术质量证据，可支撑「测试记录」交付与 Prompt 迭代。

三层评测架构

┌─────────────────────────────────────────────────────────┐ │ 第一层：Code Evaluator（确定性 · 必过） │ │ 身份门禁 · 引用格式 · 跨库拦截 · reset 清空 · 异常不崩溃 │ └───────────────────────────┬─────────────────────────────┘ │ ┌───────────────────────────▼─────────────────────────────┐ │ 第二层：Reference Match（事实题 · ground truth） │ │ 费用/时间/地点/班型名 与素材 A/B/C 标准答案比对 │ └───────────────────────────┬─────────────────────────────┘ │ ┌───────────────────────────▼─────────────────────────────┐ │ 第三层：LLM-as-Judge（语义质量 · 辅助） │ │ 推荐理由是否对应约束 · Faithfulness · 有无编造 │ └─────────────────────────────────────────────────────────┘

LangSmith 接入示例（Python）

from langsmith import Client from langsmith.evaluation import evaluate from openevals.llm import create_llm_as_judge from openevals.prompts import CORRECTNESS_PROMPT # 被测对象：Agent API def target(inputs: dict) -> dict: resp = call_agent_api( message=inputs["user_message"], session_id=inputs.get("session_id"), ) return {"text": resp["reply"], "meta": resp["trace"]} # Code Evaluator：身份门禁 def identity_gate(run, example): ref = example["reference_outputs"] if ref.get("expect_no_recommend"): hit = any(c in run.outputs["text"] for c in KNOWN_COURSES) return {"key": "identity_gate", "score": 0 if hit else 1} return {"key": "identity_gate", "score": 1} # LLM-as-Judge：语义正确性（辅助） correctness_judge = create_llm_as_judge( prompt=CORRECTNESS_PROMPT, model="openai:gpt-4o-mini", ) # 跑实验：对比 Prompt/模型版本，导出报告作测试证据 evaluate( target, data="course-agent-official-25", evaluators=[identity_gate, correctness_judge], experiment_prefix="v1-smoke", )

与状态机架构的配合

状态机负责

  • 第一层 Code Evaluator 可直接读 session 字段断言

LangSmith 负责

  • 端到端输出是否符合 25 条验收
  • Prompt/模型版本对比与回归
  • 导出 Experiment 报告 → 测试记录证据

支撑交付物「测试记录」

交付要求 评测平台如何支撑
不少于 10 组测试 从 Dataset 选取 ≥10 条 Example 跑 Experiment

注意事项

  • 不能替代评委实测：LangSmith 是开发与回归工具，官方 25 条仍需手工验证通过
  • LLM-as-Judge 不能单独判事实：费用、时间等以素材 ground truth + Code Evaluator 为准
  • 密钥规范：LangSmith API Key、Judge 模型 Key 走环境变量，禁止写入仓库
  • 网络环境：国内需确认 LangSmith 云服务可达；不可达时用 Langfuse 自托管或 pytest + JSON 报告兜底
  • 非必做加分项：需求未强制评测平台，但可体现「可稳定复现」的技术质量
实施优先级：Phase 4 先将官方 25 条录入 Dataset → 实现 8 个 Code Evaluator → 跑通 smoke experiment → 导出 ≥10 组测试记录。LLM-as-Judge 与 AgentEvals 作为迭代期增强，不阻塞主链路交付。
